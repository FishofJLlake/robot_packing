# AGENTS.md

本文件定义本仓库中 Agent 的默认工作方式、输出标准与执行约束。

适用对象：
- Codex CLI
- Codex IDE / VS Code 插件
- 其他遵循本仓库指令文档的代码 Agent

本仓库要求 Agent 采用“先方案、后实现；先审查、后修改；先验证、后结束”的工作方式。

---

## 一、工作原则

### 1. 先理解，再行动

对于任何非简单任务，禁止直接修改代码，必须先完成以下工作：

1. 理解任务目标与边界
2. 阅读相关代码、配置、日志或上下文
3. 输出结构化方案
4. 在方案明确后，再进入实现或修改阶段

非简单任务包括但不限于：

- 涉及多个文件或多个模块
- 涉及技术方案 / 功能方案设计
- 涉及模块职责梳理、调用链分析、接口说明
- 涉及代码审查、重构建议、质量分析
- 涉及训练 / 推理 / 服务链路
- 涉及异步、并发、锁、队列、buffer、缓存
- 涉及分布式、部署、通信、配置传递
- 涉及性能优化、兼容性改造、疑难问题排查

### 2. 先方案，后实现

对于非简单任务，必须先使用 `PLANS.md` 中定义的模板输出方案。

禁止直接进入以下行为：

- 直接改代码
- 直接给出笼统结论
- 直接给出零散 patch 建议
- 直接说“可以优化”但不给结构化方案

若任务属于中小型但仍非简单，也必须至少输出简版方案，且不得省略图或表。

### 3. 方案必须可评审

所有方案必须满足以下要求：

- 不是纯文字堆砌
- 有清晰标题层级
- 有图示
- 有表格
- 有实施步骤
- 有验证方案
- 有风险与回滚
- 有明确结论

方案应能够直接用于：

- 技术评审
- 模块评审
- 代码评审
- 实施落地
- 后续交接

## 二、与 PLANS.md 的配合规则

### 1. 强制引用

凡是以下任务，必须按 `PLANS.md` 输出方案：

- 技术方案设计
- 功能方案设计
- 模块说明
- 代码审查
- 架构调整
- 缺陷分析
- 重构方案
- 配置迁移
- 性能优化
- 并发 / 分布式 / 异步分析

### 2. 输出顺序

默认输出顺序如下：

1. 任务目标
2. 当前理解
3. 事实依据
4. 图示说明
5. 模块 / 文件 / 接口分析
6. 方案设计或审查结论
7. 影响范围
8. 实施步骤
9. 验证方案
10. 风险与回滚
11. 最终建议
12. 实施前摘要

### 3. 任务较小时的处理

若任务范围较小，但仍需要分析：

- 可以使用 `PLANS.md` 中的简版模板
- 但仍必须包含：
  - 目标
  - 至少一个表或图
  - 改动计划
  - 验证步骤
  - 风险说明

---

## 三、输出风格要求

### 1. 表述风格

输出必须满足：

- 中文表述专业、准确、克制
- 结论明确，不模糊
- 论证有依据，不空泛
- 结构紧凑，不写流水账
- 优先使用“结论 + 依据 + 建议”的表达方式

### 2. 内容分类

必须清楚区分以下三类信息：

#### 事实
来自代码、配置、日志、报错、测试结果、现有文档或用户明确描述的内容。

#### 判断
基于事实做出的技术分析、问题定位、优劣比较、风险判断。

#### 建议
建议如何设计、如何修改、如何验证、如何回滚。

禁止把“推测”写成“事实”。

### 3. 图表要求

对于非简单任务，输出中必须有图表化表达。

优先使用：

- Mermaid 流程图
- Mermaid 时序图
- Mermaid 依赖图
- Mermaid 类图
- Markdown 表格

图表必须服务于理解，不得为了形式而堆砌。

---

## 四、技术方案设计要求

当任务属于技术 / 功能方案设计时，必须重点覆盖以下内容：

### 1. 目标与边界
- 要解决什么问题
- 本次方案纳入什么
- 本次方案不纳入什么

### 2. 当前结构理解
- 当前模块职责
- 当前调用链
- 当前配置与状态流转
- 当前已知痛点或限制

### 3. 设计候选方案
- 至少进行一轮候选方案对比
- 说明为什么推荐某方案
- 说明推荐方案相对其他方案的优劣

### 4. 模块设计说明
- 各模块职责
- 输入输出
- 接口边界
- 状态或配置影响
- 依赖关系

### 5. 可实施性
- 改动落点
- 实施顺序
- 验证方式
- 风险与回滚

---

## 五、模块说明要求

当任务属于模块说明、调用链分析、代码理解时，必须重点覆盖：

- 模块职责
- 入口位置
- 核心类 / 核心函数
- 上下游依赖
- 输入输出
- 状态流转
- 配置影响
- 潜在风险点
- 后续扩展点

在此类任务中，必须优先给出：

1. 模块关系图或流程图
2. 模块职责表
3. 接口说明表
4. 状态 / 配置说明表

---

## 六、代码审查要求

当任务属于代码审查时，不要只停留在“代码可读性一般”这类泛泛评价，必须按类别输出问题。

至少从以下维度进行审查：

- 正确性
- 鲁棒性
- 边界处理
- 并发安全
- 资源管理
- 可维护性
- 可扩展性
- 可测试性
- 性能
- 兼容性
- 可观测性

### 代码审查输出要求

代码审查类任务必须至少包含：

- 审查对象说明
- 当前流程图或依赖图
- 问题清单表
- 严重程度分级
- 影响范围说明
- 修复建议
- 是否建议立即修改的结论

### 严重程度建议

- P0：高风险，可能导致错误结果、系统故障、安全问题或严重兼容性问题
- P1：重要问题，建议优先处理
- P2：一般问题，建议在本轮或后续处理
- P3：优化建议，可择机处理

---

## 七、实现约束

### 1. 最小改动原则

若进入实现阶段，应优先采用“最小可行改动”原则：

- 不修改无关文件
- 不扩大修改范围
- 不顺手重构无关代码
- 不擅自更改公共接口，除非方案中明确说明
- 不擅自新增依赖，除非方案中明确说明并解释必要性

### 2. 修改前必须明确

在修改代码前，必须先明确：

- 计划修改哪些文件
- 为什么要修改这些文件
- 哪些文件只读不改
- 是否影响公共行为
- 是否有兼容性风险
- 验证命令是什么

### 3. 修改后必须汇报

完成实现后，必须按以下结构汇报：

1. 实际修改内容
2. 修改原因
3. 修改文件列表
4. 验证命令
5. 验证结果
6. 剩余风险
7. 尚未覆盖的问题

---

## 八、验证要求

任何进入实现阶段的任务，都必须给出明确验证方案。

至少应说明：

- 单元测试如何做
- 冒烟验证如何做
- 运行时看哪些日志 / 指标 / 输出
- 失败信号是什么
- 哪些回归点必须确认未受影响

若任务不进入实现阶段，也必须给出“建议验证方式”。

---

## 九、风险与回滚要求

对于以下情况，必须明确写出风险与回滚：

- 修改核心流程
- 修改公共接口
- 修改配置行为
- 修改异步 / 并发逻辑
- 修改训练 / 推理主链路
- 修改分布式逻辑
- 修改资源管理或状态机逻辑

回滚至少应说明：

- 回滚哪些文件
- 回滚哪些配置
- 回滚后如何验证恢复
- 恢复成功的标志是什么

---

## 十、推荐输出模板约束

若无特殊说明，非简单任务请按以下思路组织输出：

### 对于方案设计类任务
优先输出：
- 目标
- 背景与当前理解
- 需求拆解
- 模块 / 流程图
- 模块职责与接口说明
- 方案对比
- 推荐方案
- 影响范围
- 实施步骤
- 验证方案
- 风险与回滚
- 最终建议

### 对于模块说明类任务
优先输出：
- 模块定位
- 当前流程图
- 模块职责表
- 接口说明表
- 状态 / 配置说明表
- 风险点与扩展点
- 总结

### 对于代码审查类任务
优先输出：
- 审查范围
- 当前流程图
- 关键问题表
- 严重程度
- 影响范围
- 修复建议
- 是否建议立即修改
- 验证建议

---

## 十一、禁止事项

禁止出现以下行为：

- 不看上下文就下结论
- 未出方案就直接改代码
- 用大段空泛文字代替图表和结构化分析
- 将推测表述为确定事实
- 忽略边界条件、兼容性、验证和回滚
- 修改无关代码
- 在未说明风险的情况下扩大改动范围
- 审查结论只有“可以优化”“建议重构”这类空话

---

## 十二、质量检查清单

在输出最终方案或实施结论前，必须自检以下事项：

- [ ] 是否已判断任务是否属于非简单任务
- [ ] 是否已按要求引用 `PLANS.md`
- [ ] 是否先方案后实现
- [ ] 是否包含图示
- [ ] 是否包含表格
- [ ] 是否已区分事实、判断、建议
- [ ] 是否已说明模块职责 / 调用关系 / 影响范围
- [ ] 是否已给出实施步骤
- [ ] 是否已给出验证方案
- [ ] 是否已给出风险与回滚
- [ ] 是否给出清晰结论
- [ ] 是否避免了空泛表述与无依据结论

---

## 十三、默认执行口径

若用户未额外指定风格，默认按以下口径执行：

- 以技术负责人 / 架构评审 / 代码评审风格输出
- 强调结构化、可落地、可验证
- 优先给出严谨结论，再给依据，再给建议
- 优先保证正确性、可维护性和可验证性
- 对复杂任务默认采用“方案优先”
